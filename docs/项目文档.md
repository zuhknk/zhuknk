# LaienTech iOS App Review Analyzer — 项目文档

## 项目概述

本项目的目标是为 **LaienTech 面试评估** 构建一个完整的 iOS App 评论分析工具。用户输入 App Store 链接或上传 JSON/CSV 评论文件后，系统通过 8 阶段 SSE 流式管道完成数据采集、清洗、分析和输出，最终生成分析发现、产品需求（PRD）、测试用例和校验报告。

技术栈：**Vue 3.5 + Pinia 2.3 + Vite 6**（前端） + **Python 3.10+ / FastAPI / SSE**（后端） + **OpenAI GPT-4o-mini**（LLM 分析，可选）。

## 架构设计

### SSE 8 阶段管道

整个分析流程通过 **Server-Sent Events** 实时推送，前端使用 `fetch` + `ReadableStream` 解析。

```
POST /api/analyze
  │
  ├─ parsing(5%)     → 解析请求参数，提取 App ID，确定分析范围
  ├─ collecting(10%) → 采集评论（RSS API 或文件导入）
  ├─ cleaning(35%)   → 精确去重 + 模糊去重 + 噪声过滤 + 语言检测
  ├─ analyzing(50%)  → LLM 主题发现（有 API Key）/ 统计模式（无 API Key）
  ├─ evidence(65%)   → 证据验证（不足 3 条支撑则降级）
  ├─ prd(75%)        → 生成产品需求文档
  ├─ testcase(85%)   → 生成可追溯的测试用例
  ├─ validating(95%) → 校验可追溯性和一致性
  └─ done            → 返回完整结果 JSON
```

### 数据模型（Pydantic）

| 模型 | 关键字段 | 用途 |
|------|----------|------|
| `AnalysisRequest` | `app_store_url`, `file_data`, `file_format`, `analysis_goal` | 请求入口 |
| `ReviewRaw` | `review_id`, `rating`, `title`, `content`, `version`, `vote_sum` | 原始评论 |
| `ReviewCleaned` | 继承 ReviewRaw，增加 `is_duplicate`, `is_noise`, `language` | 清洗后评论 |
| `ReviewFinding` | `finding_id`, `topic`, `severity`, `sentiment`, `review_ids`, `evidence_level`, `source`, `uncertainty` | 分析发现 |
| `Requirement` | `req_id`, `title`, `user_story`, `priority`, `related_review_ids`, `acceptance_criteria` | PRD 需求 |
| `TestCase` | `case_id`, `req_id`, `steps`, `expected_result`, `source_review_ids` | 测试用例 |
| `ValidationReport` | `validation_passed`, `issues`, `traceability`, `summary` | 校验报告 |

### 追溯性设计

- `TestCase.source_review_ids` → 追溯到具体评论，回答"这个测试用例是基于哪些用户反馈生成的"
- `Requirement.related_review_ids` → 追溯到评论，回答"这个需求来自哪些用户声音"
- `ReviewFinding.review_ids` → 追溯到评论，回答"这个发现由哪些评论支撑"

### 双模式运行

- **LLM 模式**（已配置 API Key）：GPT-4o-mini 驱动主题发现、PRD 生成、测试用例生成、校验
- **统计模式**（未配置 API Key）：基于规则生成评分分布、语言分布等统计发现，所有结果标记 `source: "stats"`

### 组件功能

- **InputPanel** — 双模式输入（URL / 文件导入），支持 JSON/CSV 解析，支持 `analysis_goal` 输入
- **ProgressPipeline** — 8 阶段进度条实时更新，完成后展示评论/发现/需求/测试用例 4 个标签页，含校验报告和警告信息

- **TestCasesPanel** — 前置条件/步骤/预期结果，`source_review_ids` 追溯链接

核心状态管理，管理 SSE 连接、8 阶段进度、结果数据、过滤和导出功能。使用 `fetch` + `ReadableStream` 手动解析 SSE 事件流（EventSource 不支持 POST 请求）。

## 后端详情

### 关键实现

**采集层** (`collectors/app_store.py`)
- `parse_app_store_url()` — 正则提取 country 和 app_id，支持 US/CN 等多国 URL
- `collect_reviews()` — 异步分页调用 iTunes RSS API，`tenacity` 重试 3 次
- `collect_reviews_from_file()` — 从 JSON/CSV 数据构建 ReviewRaw 列表

**清洗层** (`cleaners/review_cleaner.py`)
- 精确去重：`review_id` 去重
- 模糊去重：前 50 字符 + rating 指纹去重
- 噪声检测：空内容、少于 5 字符、纯 emoji
- 语言检测：`langdetect` 库

**LLM 层** (`llm_client.py`)
- 单例 `AsyncOpenAI` 客户端
- `analyze_batch()` — 3 次重试，指数退避，`response_format: json_object` 强制结构化输出
- `format_reviews_for_llm()` — 提取关键字段减少 token 消耗

**分析层** (`analyzers/review_analyzer.py`)
- 分批处理（每批 40 条），跨批次主题合并
- 证据验证：< 3 条 → `insufficient`，< 5 条 → `limited`，>= 5 条 → `sufficient`
- 置信度降级：证据不足时自动降低 `confidence`

**生成层** (`generators/prd_generator.py`, `generators/testcase_generator.py`)
- 每条需求/测试用例自动关联 `source_review_ids`/`related_review_ids`
- 支持 `analysis_goal` 注入提示词，聚焦分析范围

**校验层** (`validators/result_validator.py`)
- LLM 校验：检查需求-测试用例覆盖率和一致性
- 统计模式回退：`generate_placeholder()` 生成评分分布 + 语言分布

### 配置项

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `RSS_MAX_PAGES` | 10 | 最多采集 10 页（500 条） |
| `BATCH_SIZE` | 40 | LLM 每批处理评论数 |
| `MIN_EVIDENCE_COUNT` | 3 | 最少证据数阈值 |
| `LLM_MODEL` | `gpt-4o-mini` | LLM 模型 |
| `REVIEW_MIN_LENGTH` | 5 | 最短有效评论字符数 |

## 开发过程与问题解决

### 1. 交接文档修复

从交接文档中发现了 15 处错误：代码导入缺失、配置不一致、emoji 正则表达式错误、以及 3 个完全缺失的模块（analyzers、generators、validators）。修复后重新实现了所有缺失模块。

### 2. 中文路径问题

项目初始创建在 `e:\面试\app-review-analyzer\`，npm 无法处理中文路径。将项目迁移至 `e:\object_miansi\app-review-analyzer\` 解决。

### 3. 评估需求补充

前端初始版本缺少三项要求：`analysis_goal` 输入框、CSV 文件导入、测试用例 `source_review_ids` 追溯。全部补充完成，API 文档同步更新。

### 4. iTunes RSS Feed API 已废弃

**关键发现**：Apple 的 iTunes RSS Feed API（`itunes.apple.com/{country}/rss/customerreviews/`）对所有测试 App 返回 0 条评论。测试了 9 个主流 App（Instagram、YouTube、Facebook、TikTok、Reddit、WhatsApp、Twitter、Viber、Workout for Women），全部返回空结果。这不是 URL 格式或网络问题，而是 Apple 服务端已停止提供评论数据。

**处理方案**：后端在 RSS 返回空数据时正确返回 `error` 事件，提示用户使用文件导入模式。

### 5. 统计模式 SSE 阶段补全

初始版本在统计模式下跳过了 evidence、prd、testcase、validating 阶段的 SSE 推送。修复后统计模式也完整推送 8 个阶段，确保前端进度条体验一致。

## 演示建议

由于 RSS API 已不可用，评估演示建议使用**文件导入模式**：

1. 打开前端 `http://localhost:3000`
2. 切换到"文件导入"选项卡
3. 上传 `test_reviews.json`（或拖拽任意 JSON/CSV 评论文件）
4. 输入 `analysis_goal`：如 `关注女性健身应用的可用性、崩溃和订阅问题`
5. 点击"开始分析"，观察 8 阶段 SSE 管道实时推送
6. 完成后查看评论/发现/需求/测试用例 4 个标签页，验证追溯性
